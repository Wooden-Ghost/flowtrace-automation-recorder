# FlowTrace 自动化流程留痕系统

> 面向网页自动化脚本面板的流程记录、周期分析与动作包沉淀方案。  
> 不是鼠标宏，不是录屏工具，也不是普通日志系统。FlowTrace 解决的是：**自动化脚本如何从“很多按钮”升级为“可复用、可分析、可维护的流程”。**

---

## 1. 直接结论

网页自动化脚本发展到一定阶段后，真正的问题不再是“缺一个按钮”，而是：

1. 人工、脚本、网页动作混在一起，出错后难复盘。
2. 很多按钮本质上是重复动作包，但没有被识别和沉淀。
3. 同一流程会反复执行多轮，但每一轮的差异、耗时、异常没有结构化记录。
4. 脚本经验难迁移：一个项目里调出来的等待、重试、判断条件，不能自然复用到另一个项目。
5. AI 想辅助改脚本，但拿到的往往是截图、日志碎片和人工描述，缺少结构化证据。

**FlowTrace 的价值：把脚本运行过程变成可分析的数据，把重复流程提取成动作包，把不稳定步骤暴露出来。**

最终目标不是“录制后直接回放”，而是形成一条开发链路：

```text
人工/脚本/网页动作记录
→ 周期识别
→ 重复片段分析
→ 稳定性判断
→ 动作包候选
→ 正式自动化脚本
```

---

## 2. 项目定位

**FlowTrace｜自动化流程留痕系统**

一句话说明：

```text
FlowTrace 记录网页自动化中人工、脚本、页面共同完成的流程，识别重复周期，分析稳定性，并输出可复用动作包。
```

更准确的定位：

```text
FlowTrace 是自动化开发的“过程证据层”。
它不替代具体业务脚本，也不替代 RPA。
它负责记录和分析“一个自动化流程到底是怎么跑起来的、哪里稳定、哪里不稳定、哪些步骤值得固化”。
```

适用范围：

```text
Snov.io 导入/导出
CRM 联系人收集/跟进
Outlook 自动发送
后台表格下载/上传
网页批量录入
抖音/电商/管理后台类网页自动化
任何带脚本面板、人工介入、重复流程和页面状态判断的自动化项目
```

---

## 3. 需要解决的核心问题

| 问题 | 现象 | FlowTrace 要给出的结果 |
|---|---|---|
| 按钮越来越多 | 面板复杂，按钮之间依赖人工记忆 | 提取按钮背后的动作包 |
| 流程重复但未结构化 | 每轮都差不多，但没有周期记录 | 按周期记录每一轮执行过程 |
| 出错难定位 | 只知道“卡住了”，不知道卡在哪一步 | 标出异常步骤、耗时、重试、页面状态 |
| 脚本经验难迁移 | 不同项目各写各的 | 统一事件规范，交给同一个 Python 分析器 |
| AI 难以准确改脚本 | 输入是碎片截图和日志 | 输出 events.jsonl / flow_candidate.json 给 AI 分析 |

---

## 4. 系统架构

FlowTrace 不应该把复杂分析塞进油猴脚本。浏览器端越重，越难调试，也越容易干扰页面。

推荐架构：

```text
网页脚本端：轻量记录器
    只记录关键事件，不做复杂分析
        ↓
统一事件规范：ARS-v0.1
    不同项目按同一格式导出
        ↓
Python 分析端：周期分析器
    识别周期、重复片段、稳定性、异常步骤
        ↓
报告与动作包
    给人看、给 AI 看、给程序继续用
```

职责边界：

| 模块 | 负责什么 | 不负责什么 |
|---|---|---|
| 网页脚本 | 记录点击、输入、URL 变化、脚本步骤、周期标记 | 不做复杂聚类，不生成最终判断 |
| Python | 周期切分、相似度、异常排行、动作包候选、报告生成 | 不直接操作网页 |
| AI | 根据结构化结果辅助归纳、改写脚本、提出动作包设计 | 不直接控制页面，不凭空猜 DOM |

## 5. 事件规范：ARS-v0.1

不同项目的脚本都要按同一种格式导出事件。否则 SN、CRM、Outlook 各记各的，Python 后续无法统一分析。

### 5.1 基础结构

```json
{
  "schema": "ARS-v0.1",
  "project": "SN",
  "script": "SN导入导出助手",
  "script_version": "4.0.3",
  "session_id": "2026-05-14_170022",
  "events": []
}
```

### 5.2 单条事件

```json
{
  "event_id": 12,
  "time": "2026-05-14 17:03:21",
  "cycle_id": "cycle_001",
  "source": "script_auto",
  "type": "click",
  "target_text": "上传文件",
  "target_role": "button",
  "page_state": "file_import_modal",
  "url": "https://app.snov.io/prospects",
  "result": "success",
  "duration_ms": 820,
  "note": ""
}
```

### 5.3 source 必须区分

```text
human_panel   人工点击脚本面板
human_web     人工点击网页
script_auto   脚本自动动作
system        系统观察事件，比如 URL 变化、loading 消失、弹窗出现
```

这个字段很关键。它决定后续能不能判断：

```text
哪些动作可以自动化
哪些动作仍然需要人工
哪些问题是脚本导致的
哪些问题是网页自然状态变化导致的
```

### 5.4 type 建议枚举

```text
click
input
change
url_change
cycle_start
cycle_end
script_step_start
script_step_end
wait_start
wait_end
error
pause
resume
```

第一版不要贪多。字段少但稳定，比字段多但乱更有价值。

## 6. 快照是什么

快照不是截图，也不是完整 DOM。

在 FlowTrace 里，**快照是某一刻页面状态的摘要**，用于判断动作前后是否符合预期。

例如点击“上传文件”前：

```json
{
  "snapshot_type": "before_action",
  "page_state": "file_import_modal",
  "visible_buttons": ["取消", "上传文件"],
  "has_modal": true,
  "has_loading": false,
  "url": "https://app.snov.io/prospects",
  "keywords": ["从文件导入", "上传文件"]
}
```

点击后：

```json
{
  "snapshot_type": "after_action",
  "page_state": "import_wizard_mapping",
  "visible_buttons": ["取消导入", "下一步"],
  "has_modal": false,
  "has_loading": false,
  "url": "https://app.snov.io/import/prospects",
  "keywords": ["字段映射", "下一步"]
}
```

快照只回答三个问题：

```text
动作前是不是该做这一步？
动作后有没有进入预期状态？
失败时页面大概卡在哪里？
```

不需要保存完整 HTML。目标元素附近 HTML 可以保留 500～1000 字符作为排错材料，超出就截断。

## 7. 周期识别：先人工标记，再自动辅助

周期识别不要一开始就完全依赖算法。最稳的做法是：

```text
人工标记周期开始/结束
Python 按 cycle_id 分组
没有标记时，再用相似度辅助判断
```

### 7.1 人工标记

面板可以提供：

```text
开始录制
周期开始
周期结束
停止录制
导出
```

记录示例：

```json
{"type":"cycle_start","cycle_id":"cycle_001","label":"珠宝包装岗位"}
{"source":"human_panel","type":"click","target_text":"搜索"}
{"source":"script_auto","type":"click","target_text":"添加潜在客户"}
{"source":"human_web","type":"click","target_text":"选择文件"}
{"type":"cycle_end","cycle_id":"cycle_001","result":"success"}
```

人工标记优先级最高。这样第一版就能稳定分析，不必一上来做复杂自动识别。

### 7.2 自动识别

Python 后续可以把动作简化成 token：

```text
human_panel:搜索
script_auto:添加潜在客户
script_auto:从文件导入
human_web:选择文件
script_auto:上传文件
script_auto:下一步
script_auto:导入
```

如果多段序列相似度超过 80%，就标记为同类周期候选。

### 7.3 周期稳定性

每轮周期需要统计：

```text
步骤是否完整
顺序是否一致
每步耗时是否波动大
是否出现重试
是否出现异常弹窗
是否需要人工介入
是否成功结束
```

输出判断：

| 判断 | 标准 | 后续动作 |
|---|---|---|
| 稳定 | 多轮步骤一致，失败少，耗时波动小 | 可固化为动作包 |
| 基本稳定 | 主流程一致，但个别步骤需重试 | 可固化，但保留重试和超时 |
| 不稳定 | 步骤顺序变化大或失败频繁 | 继续侦察，不急着写正式自动化 |
| 危险 | 容易误点、误提交、状态不明确 | 不建议自动执行 |

## 8. 动作包沉淀

动作包不是简单把多个按钮合并。可靠动作包必须同时包含：

```text
前置条件
动作本身
后置验证
失败处理
是否允许重试
是否需要人工介入
```

示例：

```json
{
  "name": "打开文件导入",
  "stability": "high",
  "steps": [
    {
      "action": "click_by_text",
      "text": "添加潜在客户",
      "precondition": "main_page_ready",
      "postcondition": "menu_contains_从文件导入"
    },
    {
      "action": "click_by_text",
      "text": "从文件导入",
      "postcondition": "file_import_modal_open"
    }
  ]
}
```

一个片段是否值得成为动作包，看三个指标：

```text
重复出现次数
成功率
异常处理是否明确
```

不是重复就一定合并。比如“验证邮箱”可能经常出现，但它依赖按钮是否可点击、列表是否有待验证邮箱，因此更适合作为条件分支，而不是硬合并进固定动作包。

## 9. 输出结果应该长什么样

FlowTrace 的输出必须同时服务三类对象：

```text
人：快速看出流程哪里稳、哪里不稳
Python：继续统计和分析
AI：辅助归纳动作包、改正式脚本
```

推荐导出一个 zip：

```text
FlowTrace_SN_2026-05-14_170022.zip
```

包含：

```text
report.html              人看的报告
summary.json             会话摘要
events.jsonl             原始事件流
cycles.csv               周期摘要表
cycles.jsonl             周期结构数据
flow_candidate.json      动作包候选
raw_log.txt              兜底原始日志
```

### 9.1 report.html 重点展示

不要做花哨报告，只保留能帮助判断的内容：

```text
流程总览
周期列表
异常步骤排行
步骤稳定性
动作包建议
每轮详情
```

示例：

```text
流程名称：SN 导入流程
周期数量：12
成功周期：10
失败周期：2
平均耗时：48.2 秒
最高风险步骤：点击上传文件
人工介入点：选择文件
结论：主流程可固化，但上传后跳转必须保留重试
```

### 9.2 cycles.csv 用于快速筛查

```text
轮次,目标,结果,耗时,重试,异常,判断
1,珠宝包装岗位,成功,43s,0,无,稳定
2,美妆采购岗位,成功,47s,0,无,稳定
3,鞋服包装岗位,成功,66s,1,上传后未跳转,基本稳定
4,礼品高管岗位,失败,90s,3,上传按钮点击无效,不稳定
```

### 9.3 flow_candidate.json 给 AI 和正式脚本用

```json
{
  "flow_name": "SN导入一轮",
  "confidence": 0.86,
  "recommendation": "can_package_with_retry",
  "steps": [
    {
      "name": "点击添加潜在客户",
      "action": "click_by_text",
      "text": "添加潜在客户",
      "precondition": "main_page_ready",
      "postcondition": "menu_contains_从文件导入"
    },
    {
      "name": "等待人工选择文件",
      "action": "wait_button_enabled",
      "text": "上传文件",
      "manual_required": true
    }
  ],
  "warnings": [
    "文件选择无法由浏览器脚本自动完成",
    "上传文件步骤需要保留重试"
  ]
}
```

## 10. 安全边界：保护网页账号，而不是保护浏览器

浏览器卡顿可以重启，脚本写坏可以重写。真正要控制的是账号风险。

FlowTrace 的原则：

```text
记录器只观察，不主动额外操作网页。
```

允许：

```text
监听点击
监听输入
监听 URL 变化
低频记录弹窗、loading、按钮状态
记录脚本自身动作
停止后导出本地文件
```

不允许：

```text
录制器额外点击网页
录制器自动提交数据
录制器向外部服务器上传数据
录制器高频扫描完整 DOM
录制器绕过网页限制
```

账号风险主要来自额外业务行为，不是本地监听本身。FlowTrace 必须保证：**记录器不制造额外点击、不制造额外请求、不改变网页数据。**

## 11. 最小可行版本

第一版不要做成大平台。

### 浏览器脚本 v0.1

只做：

```text
开始录制
周期开始
周期结束
停止录制
导出
```

记录：

```text
面板点击
网页点击
网页输入
URL 变化
脚本自动步骤
错误日志
```

导出：

```text
events.jsonl
summary.json
raw_log.txt
```

### Python 分析器 v0.1

只做：

```text
读取 events.jsonl
按 cycle_id 分组
输出每轮动作列表
统计每轮耗时
统计动作出现次数
计算周期相似度
生成 report.html
生成 cycles.csv
```

第二阶段再加：

```text
重复片段提取
异常排行
动作包候选
稳定性评分
flow_candidate.json
```

这个顺序最稳。第一版先证明“记录不断、周期清楚、报告能看出问题”。

## 12. 参考体系，只保留对项目有用的部分

| 体系 | FlowTrace 借鉴什么 | 不照搬什么 |
|---|---|---|
| Task Mining | 记录人工如何完成任务 | 不做企业员工行为分析平台 |
| Process Mining | 用事件流分析流程、周期、瓶颈 | 不套复杂企业流程系统 |
| XES | case/event/timestamp/activity 思想 | 不使用复杂 XML 标准 |
| BPMN | 给人看的流程图表达 | 不作为底层数据格式 |
| OpenTelemetry Trace/Span | 会话-周期-步骤的层级结构 | 不做服务链路监控系统 |

这些参考体系的作用是提供成熟表达，不是给项目堆概念。FlowTrace 只拿有用的部分：

```text
事件流
周期
层级结构
流程图
动作包候选
```

## 13. 开发优先级

| 优先级 | 内容 | 完成标准 |
|---|---|---|
| P0 | ARS-v0.1 事件格式 | SN、CRM、Outlook 都能按同一结构记录 |
| P0 | 浏览器轻量记录器 | 能记录面板点击、网页点击、URL 变化、周期标记 |
| P0 | Python 周期分组 | 能按 cycle_id 输出每轮动作 |
| P1 | report.html | 人能看出哪一轮失败、哪一步不稳定 |
| P1 | cycles.csv | Excel 能快速筛查周期结果 |
| P1 | 动作相似度 | 能判断两轮是否 80% 相似 |
| P2 | flow_candidate.json | 能输出动作包候选 |
| P2 | AI 分析提示词 | AI 能根据结构化文件提出脚本修改建议 |

第一阶段只做 P0 + 一部分 P1。P2 等前面数据跑通后再做。

## 14. 最终价值

FlowTrace 对自动化开发的改变是：

```text
从“我感觉这里要加等待”
变成
“第 3、7、9 轮都卡在上传后跳转，平均等待 12 秒仍失败，所以这里必须加重试和后置验证。”
```

从：

```text
这个按钮帮我跑一段
```

变成：

```text
这段流程在 10 个周期中 9 次一致，可以沉淀成动作包。
```

从：

```text
截图 + 一堆日志 + 人工描述
```

变成：

```text
events.jsonl + cycles.csv + flow_candidate.json
```

这就是 FlowTrace 的核心价值：**让自动化开发从经验调试，转向基于事件证据的流程沉淀。**

---

## 15. 极简总结

FlowTrace 不是为了多做一个脚本工具，而是为了把“脚本开发过程”本身记录下来。

它解决三个问题：

```text
记录不断链：人、脚本、网页动作都能被统一记录。
周期可分析：重复流程能被识别，稳定和异常能被量化。
动作可沉淀：稳定片段能转成动作包，供正式自动化脚本复用。
```

它最适合的落地方式：

```text
油猴脚本轻量记录
Python 负责分析
AI 辅助归纳和改脚本
```

第一版只要做到：**能记录、能分周期、能生成可读报告**。  
不要一开始做大平台。先让一次真实脚本流程被完整记录和复盘，项目价值就成立。
